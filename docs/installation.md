# Installation

End-to-end recipe to get a server running on your main workstation
and a `ph0sphor-client` running on a VAIO P (or any Linux box of
similar shape). Aim for "binary in, paired in five minutes, dies if
the network does."

## 0. Choose your release

Tagged releases live on the
[GitHub releases page](https://github.com/akberdin/ph0sphor/releases).
The current release is **v0.1.0** and publishes:

```text
ph0sphor-v0.1.0-linux-x86_64.tar.gz       server + client + ctl
ph0sphor-v0.1.0-linux-i686.tar.gz         client only (VAIO P)
ph0sphor-v0.1.0-windows-x86_64.zip        server + ctl
ph0sphor-v0.1.0-macos-x86_64.tar.gz       server + ctl (Intel)
ph0sphor-v0.1.0-macos-arm64.tar.gz        server + ctl (Apple Silicon)
```

Each archive ships next to a `.sha256` file generated by `sha256sum`
(or `Get-FileHash` on Windows). Always verify:

```sh
sha256sum -c ph0sphor-v0.1.0-linux-x86_64.tar.gz.sha256
```

Build from source instead with `cargo build --release --workspace` if
your target isn't covered by the release matrix.

## 1. Install the server

Pick the right archive for your workstation and follow the matching
section under [`packaging/`](../packaging/):

- Linux: [`packaging/linux/README.md`](../packaging/linux/README.md)
- Windows: [`packaging/windows/README.md`](../packaging/windows/README.md)
- macOS: [`packaging/macos/README.md`](../packaging/macos/README.md)

The common minimum is:

1. Drop `ph0sphor-server` and `ph0sphorctl` onto the PATH.
2. Copy `examples/server.toml` to your config location (e.g.
   `/etc/ph0sphor/server.toml`) and adjust `[server].name`,
   `[security].token_store`, etc.
3. Start the server. On Linux the bundled hardened systemd unit at
   `packaging/linux/ph0sphor-server.service` is the recommended path.

## 2. Try `--demo` before pairing a real client

Both binaries support `--demo`. Run them in two shells:

```sh
ph0sphor-server --demo &
ph0sphor-client --demo
```

The client shows synthetic CPU/RAM/disk/network plus the seeded
demo mail and weather payloads. This proves the binaries work on
your machine without touching network configuration. Press `Tab` to
walk the six screens; `Q` exits cleanly.

To prepare external-input files for the real mail/weather collectors:

```sh
ph0sphorctl gen-demo --dir /var/lib/ph0sphor/demo-data
# wrote /var/lib/ph0sphor/demo-data/mail.json
# wrote /var/lib/ph0sphor/demo-data/weather.json
```

Then point `collectors.mail.source` and `collectors.weather.source`
at those paths in your server config.

## 3. Install the client on the VAIO P

Grab `ph0sphor-v0.1.0-linux-i686.tar.gz` onto the VAIO. It's a
single static binary plus example configs.

```sh
sudo install -Dm755 ph0sphor-client /usr/local/bin/ph0sphor-client
install -Dm640 examples/client.toml ~/.config/ph0sphor/client.toml
$EDITOR ~/.config/ph0sphor/client.toml   # set client.server, theme, etc.
```

Optional: autostart on TTY1 via the bundled systemd unit. See
[`docs/vaio-p-client.md`](vaio-p-client.md) for the full appliance
recipe, including font and layout recommendations for the 1600×768
panel and an OpenRC/Alpine alternative.

## 4. Pair the client

First launch with an empty token. The TUI shows a PAIRING banner
with a short code like `ABCD-1234`:

```text
+- PAIRING ----------------------------+
|                                      |
|  PAIRING REQUIRED                    |
|                                      |
|  CODE: ABCD-1234                     |
|                                      |
|  On the server host, confirm with:   |
|    ph0sphorctl pair confirm ABCD-1234|
|                                      |
|  (awaiting operator...)              |
|                                      |
+--------------------------------------+
```

On the server host, run:

```sh
ph0sphorctl pair confirm ABCD-1234
# pairing confirmed
```

The control endpoint is loopback-only by design (`127.0.0.1:7078`
default; configurable via `[server].control_bind`). The server
issues a 192-bit hex token, persists it to
`[security].token_store`, and pushes it to the waiting client over
the existing WebSocket. The client writes the token to
`[client].token_file` at mode 0600 (Unix) and reconnects.

Future restarts skip pairing entirely.

## 5. Verify

The header on every screen should now read:

```text
LINK: ONLINE   HOST: main-pc   UP: hh:mm:ss   BAT: -73% DSC   NET: wlan0 192.168.x.x
```

Cycle screens with `Tab` / `1-6` / `Shift+Tab`. Cycle themes with
`C`. Quit with `Q` or `Ctrl+C`. See
[`docs/screenshots/`](screenshots/) for what each screen should
look like.

## 6. Troubleshooting

- **Client stuck on CONNECT** — verify the workstation can be
  reached on its `[server].bind` address from the VAIO. Try
  `nc -zv main-pc 7077` from the VAIO.
- **Invalid token after server restart** — confirm
  `[security].token_store` points at a persistent path. If it
  pointed at the default in-memory store, pair again.
- **TUI is unreadable** — set `ui.ascii_fallback = true` in the
  client config; the client will redraw with `+ - |` borders and an
  ASCII gauge.
- **Server uses too much CPU** — verify
  `performance.send_deltas_only = true` (default) and the
  collector intervals match your needs. See
  [`docs/performance-budget.md`](performance-budget.md).
- **Logs leak too much** — set `RUST_LOG=warn` for the server.
  Tokens are always passed through `redact_token()` before reaching
  any logger.

For deeper context, [`docs/security-model.md`](security-model.md)
covers the threat model and the pairing flow in detail.
